# VK Helpdesk Bot для школы — техническое задание

> Бот сообщества ВКонтакте для приёма и обработки заявок по техническим неполадкам
> (компьютеры, проекторы, интернет, принтеры и т.д.). Учителя подают заявки через
> кнопочный диалог, техник/админ ведёт их по статусам, все видят актуальное состояние.

Этот документ — спецификация для сборки проекта через Claude Code. Читается сверху
вниз: сначала контекст и решения, потом конкретика (схема БД, сценарии, структура),
в конце — порядок сборки по этапам и список «подводных камней».

---

## 1. Цель и рамки

**Что делаем:** тикет-систему внутри VK. Минимально жизнеспособный продукт —
заявитель создаёт заявку через кнопки, техник получает уведомление, меняет статус,
заявитель узнаёт об изменении.

**Чего НЕ делаем на старте** (но архитектуру под это закладываем): веб-админку,
аналитику, интеграции с почтой/Telegram, SLA-таймеры. Всё это — потом.

**Масштаб:** одна школа, десятки заявок в неделю, единицы техников. Это важно —
не нужен Postgres, Redis, очереди и микросервисы. Один процесс + SQLite-файл.

---

## 2. Технические решения (и почему именно так)

| Решение | Выбор | Почему |
|---|---|---|
| Язык | Python 3.12 | Стандарт для VK-ботов, много примеров |
| Фреймворк | **vkbottle 4.7+** | Async, встроенный FSM (state dispenser), удобные клавиатуры |
| Подключение | **Bot Long Poll** | Не нужен публичный HTTPS-сервер; бот сам опрашивает VK |
| Хранилище | **SQLite через aiosqlite** | Один файл, ноль администрирования, async-совместимо |
| Конфиг | `.env` + pydantic-settings | Токен и ID админов не в коде |
| Логи | loguru | Читаемые логи из коробки |
| Менеджер пакетов | uv (или pip + venv) | Быстро; на усмотрение |

**Почему Long Poll, а не Callback API:** Callback требует публичный домен с HTTPS и
обработку подтверждения сервера. Для школы это лишняя инфраструктура. Long Poll
запускается одной командой и работает с любого VPS или даже из локальной сети.

**Почему SQLite, а не Google Sheets:** Sheets соблазнительны тем, что завуч видит
таблицу без админки, но упираются в лимиты API и медленные запросы. SQLite надёжнее
и быстрее. Если «таблица для завуча» реально нужна — закладываем экспорт в CSV/Sheets
как отдельную фичу (см. раздел 9), а не как основное хранилище.

**Важно про async:** vkbottle асинхронный, поэтому БД тоже должна быть async
(`aiosqlite`). Нельзя дёргать синхронный `sqlite3` прямо в хендлерах — заблокирует
event loop. Все обращения к БД — через `await`.

## 3. Роли пользователей

- **Заявитель** (учитель/сотрудник) — создаёт заявки, видит свои заявки и их статусы.
- **Техник/Админ** — видит все новые заявки, берёт в работу, меняет статусы,
  пишет комментарий при закрытии. Определяется по белому списку VK ID в `.env`.

Решение про учеников оставляем настраиваемым: флаг `ALLOW_ALL_USERS`. По умолчанию
`false` — заявки принимаются от всех, но если поползёт спам, можно включить режим
«только из списка разрешённых» или добавить капчу-подтверждение. На старте проще
принимать от всех и не усложнять.

## 4. Модель данных

Две таблицы. Минимум, но с заделом на рост.

### `tickets`
| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK AUTOINCREMENT | Номер заявки (его показываем людям: «Заявка #42») |
| `author_vk_id` | INTEGER | Кто подал |
| `author_name` | TEXT | Имя на момент подачи (кэш, чтобы не дёргать API) |
| `location` | TEXT | Кабинет / локация |
| `category` | TEXT | Тип: computer / projector / internet / printer / other |
| `description` | TEXT | Свободное описание проблемы |
| `status` | TEXT | new / in_progress / done / rejected |
| `assigned_to` | INTEGER NULL | VK ID техника, взявшего заявку |
| `resolution` | TEXT NULL | Комментарий при закрытии/отклонении |
| `created_at` | TEXT (ISO 8601) | Дата создания |
| `updated_at` | TEXT (ISO 8601) | Дата последнего изменения статуса |

### `status_log` (история переходов — для прозрачности и будущей аналитики)
| Поле | Тип | Описание |
|---|---|---|
| `id` | INTEGER PK | |
| `ticket_id` | INTEGER FK → tickets.id | |
| `old_status` | TEXT NULL | |
| `new_status` | TEXT | |
| `changed_by` | INTEGER | VK ID того, кто изменил |
| `changed_at` | TEXT (ISO 8601) | |

> Статусы держим строками-константами в одном модуле (`enums.py`), а не магическими
> литералами по коду. Категории — тоже там, с человекочитаемыми подписями для кнопок.

**Жизненный цикл статуса:**
```
new ──(техник берёт)──> in_progress ──(готово)──> done
 │                            │
 └────(отклонить)───> rejected └──(отклонить)──> rejected
```

## 5. Сценарии (UX)

Главный принцип: **учителя не технари — минимум свободного ввода, максимум кнопок.**
Свободный текст только там, где без него никак (описание проблемы и кабинет).

### 5.1. Подача заявки (FSM-диалог)

Пошагово через состояния vkbottle (state dispenser):

1. **Старт.** Пользователь пишет что угодно или жмёт «📝 Новая заявка» →
   стартовое меню с кнопками.
2. **Категория.** Кнопки: «💻 Компьютер», «📽 Проектор», «🌐 Интернет»,
   «🖨 Принтер», «❓ Другое». → сохраняем в state.
3. **Локация.** «Укажите кабинет/место (например, "каб. 204" или "учительская")».
   Свободный ввод. Можно дать кнопки-подсказки с частыми кабинетами + «Ввести вручную».
4. **Описание.** «Опишите, что случилось». Свободный ввод. Если прислали фото —
   принимаем вложение (см. раздел 9, на старте можно пропустить).
5. **Подтверждение.** Показываем собранную заявку целиком + кнопки
   «✅ Отправить» / «✏️ Заполнить заново» / «❌ Отмена».
6. **Готово.** Сохраняем в БД → «Заявка #N принята». Сбрасываем state.
   → уведомление всем техникам.

На каждом шаге доступна кнопка «❌ Отмена» — выход из FSM без сохранения.
Защита: если пользователь уже в процессе подачи и пишет «новая заявка» — мягко
напоминаем, что есть незавершённая, и предлагаем продолжить или начать заново.

### 5.2. Команды заявителя

- `📋 Мои заявки` — список своих заявок с номерами, категорией и статусом
  (последние N, например 10).
- `ℹ️ Помощь` — короткая инструкция.

### 5.3. Сценарий техника/админа

- `🔧 Новые заявки` — список со статусом `new`. Под каждой — кнопка «Взять в работу».
- `📂 В работе` — заявки, назначенные на этого техника (или все, настраиваемо).
- Открытая заявка показывает все поля + инлайн-кнопки действий:
  «▶️ В работу», «✅ Выполнено», «🚫 Отклонить».
- При «Выполнено»/«Отклонить» — бот просит короткий комментарий (`resolution`),
  затем меняет статус и уведомляет заявителя.

### 5.4. Уведомления (двусторонние)

- Техникам — при создании новой заявки (рассылка по списку `ADMIN_IDS`).
- Заявителю — при любой смене статуса его заявки, с текстом резолюции если есть.

> Все уведомления отправляем аккуратно: VK может ограничивать рассылки. Ловим
> ошибки отправки конкретному пользователю и не роняем процесс (один недоступный
> получатель не должен ломать рассылку остальным).

## 6. Структура проекта

```
vk-helpdesk-bot/
├── .env.example            # шаблон конфига (без секретов)
├── .gitignore              # .env, *.db, __pycache__
├── README.md               # запуск, настройка группы VK, деплой
├── pyproject.toml          # зависимости
├── requirements.txt        # дубль для pip, если без uv
└── app/
    ├── __init__.py
    ├── main.py             # точка входа: сборка Bot, регистрация blueprints, run
    ├── config.py           # pydantic-settings: TOKEN, ADMIN_IDS, ALLOW_ALL_USERS, DB_PATH
    ├── enums.py            # Status, Category + подписи и эмодзи для кнопок
    ├── db/
    │   ├── __init__.py
    │   ├── database.py     # инициализация aiosqlite, миграции (CREATE TABLE IF NOT EXISTS)
    │   └── repository.py   # CRUD: create_ticket, get_ticket, list_by_status,
    │   │                   #       update_status, list_by_author ...
    ├── keyboards/
    │   ├── __init__.py
    │   ├── user.py         # клавиатуры заявителя
    │   └── admin.py        # клавиатуры техника
    ├── states.py           # FSM-состояния подачи заявки
    ├── handlers/
    │   ├── __init__.py
    │   ├── common.py       # /start, помощь, отмена
    │   ├── create_ticket.py# FSM-сценарий подачи
    │   ├── user.py         # «мои заявки»
    │   └── admin.py        # список заявок, смена статусов
    ├── services/
    │   ├── __init__.py
    │   └── notifications.py# рассылка техникам и заявителю
    └── utils/
        ├── __init__.py
        ├── access.py       # is_admin(vk_id) по списку из конфига
        └── formatting.py   # красивый рендер заявки в текст сообщения
```

Каждый файл хендлеров — отдельный vkbottle `BotLabeler`/blueprint, собираемый в
`main.py`. Так логика не сваливается в один гигантский файл.

## 7. Конфигурация (`.env.example`)

```dotenv
# Токен сообщества (Управление → Работа с API → Ключи доступа)
VK_TOKEN=

# VK ID техников/админов через запятую
ADMIN_IDS=123456,789012

# Принимать заявки от всех (true) или только из белого списка (false)
ALLOW_ALL_USERS=true

# Путь к файлу БД
DB_PATH=helpdesk.db
```

`config.py` валидирует это через pydantic-settings: `ADMIN_IDS` парсится в
`list[int]`, отсутствие `VK_TOKEN` — падение на старте с понятной ошибкой.

## 8. Порядок сборки (для Claude Code — по этапам, каждый проверяемый)

> Строй инкрементально и проверяй после каждого этапа, не пиши всё сразу.

1. **Каркас.** Структура папок, `config.py`, `.env.example`, зависимости,
   `main.py` с пустым ботом, который отвечает «привет» на любое сообщение.
   Проверка: бот запускается и отвечает.
2. **БД.** `database.py` + `repository.py`, создание таблиц при старте, простые
   CRUD. Проверка: юнит-тест или ручной скрипт создаёт и читает заявку.
3. **Подача заявки (FSM).** `states.py`, `create_ticket.py`, клавиатуры заявителя.
   Полный сценарий до сохранения в БД. Проверка: заявка реально появляется в БД.
4. **Просмотр своих заявок.** `user.py` + рендер. Проверка: «Мои заявки» отдаёт список.
5. **Админ-функции.** `admin.py`: список новых, взятие в работу, смена статуса,
   ввод резолюции, `access.py`. Проверка: техник проводит заявку new → done.
6. **Уведомления.** `notifications.py`, подключить к созданию и смене статуса.
   Проверка: техник получает новую заявку, заявитель — смену статуса.
7. **Шлифовка.** Обработка отмены/незавершённого FSM, ошибки отправки, README.

Цель MVP — этапы 1–6. Седьмой делает продукт пригодным к реальной эксплуатации.

## 9. Идеи на потом (моё видение, не для первого релиза)

Сознательно вынесено за рамки MVP, чтобы не утонуть. Но архитектуру (особенно
`status_log` и слой `services/`) закладываем так, чтобы это добавлялось без переделки.

- **Фото к заявке.** Учителю проще сфоткать «синий экран», чем описать. Принимать
  вложения на шаге описания, сохранять VK-attachment-строку, показывать технику.
- **Экспорт для завуча.** Команда/кнопка «выгрузить в CSV» или периодический
  экспорт в Google Sheets — закрывает тот самый запрос «хочу видеть всё таблицей»,
  не делая Sheets основным хранилищем.
- **Статистика.** На базе `status_log`: сколько заявок за месяц, среднее время
  закрытия, топ проблемных кабинетов. Полезно для отчёта администрации школы.
- **Приоритеты.** Поле priority (обычная/срочная). «Проектор не работает, а через
  5 минут урок» — это не то же самое, что «мышка тугая».
- **SLA-напоминания.** Если заявка висит в `new` дольше N часов — пинговать техника.
- **Несколько техников с зонами.** Маршрутизация по категории или этажу.
- **Рейтинг/обратная связь.** После закрытия спросить, решена ли проблема на самом деле.

## 10. Подводные камни VK (учесть при реализации)

- **Настройки группы.** В сообществе нужно: включить сообщения, в «Работа с API →
  Long Poll API» поставить актуальную версию (5.131+) и галочки на событиях
  входящих сообщений. Без этого бот не получает апдейты — описать в README.
- **Клавиатуры.** У VK есть лимиты на число кнопок в ряду и рядов. Inline-кнопки
  (`inline=True`) живут внутри сообщения и удобны для действий над конкретной
  заявкой; обычные клавиатуры — для постоянного меню.
- **Имена пользователей.** Чтобы показать имя автора, нужен вызов `users.get`.
  Кэшируем имя в `author_name` при создании, чтобы не дёргать API на каждый рендер.
- **Рассылки.** Сообщения первым (когда бот пишет пользователю, который давно не
  писал) могут блокироваться правилами VK. Для уведомлений заявителю это обычно ок
  (он недавно взаимодействовал), но ошибки отправки всё равно ловим.
- **Один процесс — один Long Poll.** Не запускай две копии бота на одном токене,
  получишь конфликты апдейтов.

## 11. Деплой

Самый простой путь для школы:

- **VPS** (в РФ удобно Timeweb / Selectel, минимального тарифа хватит), Ubuntu.
- Процесс под **systemd** или в Docker, чтобы поднимался после перезагрузки и
  рестартовал при падении.
- Файл БД (`helpdesk.db`) — на диске рядом; настроить простой бэкап (копия файла
  по cron в другое место). SQLite бэкапится копированием файла.
- Альтернатива почти бесплатно: **Raspberry Pi** или старый компьютер прямо в школе.

В README дать оба варианта: «запустить локально для теста» и «развернуть на VPS
через systemd».
